## 1. 结论摘要

当前提出的 `box` 需求是合理的，而且已经接近 TimeFiles 继续扩展前必须补上的结构层能力。原因有三：

1. 现有产品以“单一全局任务树”为中心，任务量一旦增长，不同语义的任务会被挤在同一棵树里，阅读、搜索、操作成本会迅速上升。
2. 当前“任务系”概念更像是技术上的根节点分组，不足以表达“项目空间 / 分类空间 / 临时事务空间”这三种不同的使用场景。
3. 当前任务模型的 `status` 是计时状态，不是项目管理状态，因此它可以表达“是否在计时”，但不能充分表达“项目是否完成、是否排期、是否需要单独项目视图”。

但该需求不能直接以“给任务再加几种特殊标签”的方式完成。推荐结论如下：

1. `box` 必须成为一等实体（first-class entity），不能只依赖标签字符串推导。
2. `#!task:xxx`、`#!box:xxx`、`#!others` 更适合作为用户输入层或兼容层语法，而不是底层唯一存储形式。
3. 第一阶段先做“box 基础能力 + box 切换后的任务树隔离 + 新的 box 页面”。
4. “任务管理型 box 的特殊视图 a”应作为第二层需求推进，先定义最小项目视图，再决定是否真的落成甘特图。
5. 初版必须坚持“一个任务只属于一个主 box”，否则当前任务树、重挂载、统计、命令系统都会变复杂。

## 2. 现状评估

基于当前代码结构，系统现状如下：

1. 后端任务模型只包含 `id / parent_id / title / status / created_at / archived_at`，没有 `box_id`、项目状态、计划时间等字段。
2. 前端 `getOverview()` 会一次性取回所有未归档任务，任务树页面本地用 `parent_id` 组装成一棵全局树。
3. 标签系统是自由文本标签，只能表达附加属性，不能表达 `box` 的类型、完成状态、排序、隐藏规则或专属视图配置。
4. 当前命令输入仅支持 `#tag` 形式的简单标签追加，不支持 `#!task: xxx` 这种包含类型前缀、冒号和空格的结构化语法。
5. 当前“当前任务系”本质上是“当前任务所在根节点子树”，不是用户可管理、可切换的工作空间。
6. 统计页默认对全量任务统计，没有 box 维度。
7. 现有 `status=idle/running/paused/stopped` 是计时状态，不等于项目生命周期状态；因此它不能直接支撑甘特图或项目管理视图。

由此可以判断：`box` 会影响数据模型、迁移、查询接口、命令语法、树页面、主工作台、统计页面和后续项目视图设计，不是单一页面功能。

## 3. 设计目标

### 3.1 目标

1. 把原来的“单一任务树”拆分成多个可切换的工作空间。
2. 让不同用途的任务使用不同 box 承载：
   - 任务管理型 box：用于项目或长期任务组织。
   - 普通分类 box：用于普通分类归档。
   - 杂项 box：用于临时任务、快速计时、低结构负担场景。
3. 保持现有单活动上下文、事件溯源和父子任务计时语义不被破坏。
4. 允许用户在“切换 box”时同时切换任务树上下文。
5. 提供一个新的 box 页面，集中完成 box 浏览、切换和管理。

### 3.2 非目标

1. 不在 box 第一阶段引入多活动计时上下文。
2. 不在第一阶段引入多人协作。
3. 不在第一阶段直接承诺完整甘特图。
4. 不在第一阶段支持“一个任务同时属于多个 box”。

## 4. 核心概念与关键决策

### 4.1 Box

`box` 是任务的主归属空间，用来承载任务树切换、工作空间隔离和后续专项视图。

建议字段：

- `id`
- `name`
- `box_type`
- `status`
- `sort_order`
- `created_at`
- `completed_at`
- `archived_at`

### 4.2 Box 类型

建议初版定义三种类型：

1. `project`
   - 对应“任务管理型 box”。
   - 行为上等同于普通 box，但额外拥有独立右侧视图 a。
2. `category`
   - 对应“普通分类 box”。
   - 主要用于把任务树按主题、领域、职责或生活场景拆开。
3. `others`
   - 对应“杂项 box”。
   - 用于承接临时任务和快速计时场景。
   - 推荐作为系统内置 box，而不是普通用户随意创建多个同类型匿名 box。

### 4.3 Box 绑定规则

推荐初版采用以下强约束：

1. 一个任务仅能绑定一个主 box。
2. 一个子树内的所有任务必须属于同一个 box。
3. 创建子任务时默认继承父任务的 box。
4. 创建根任务时必须落入某个当前 box；若用户未显式选择，则使用当前活跃 box 或默认系统 box。
5. 跨 box 的移动必须是显式操作，不能通过普通 `reparent_task` 隐式完成。
6. 原先的所有未进行box分类任务直接迁移到一个box，例如“未分箱”，而不是直接迁移到 `others`

这样做的原因：

1. 当前产品的核心结构仍然是树。若一个任务属于多个 box，会直接导致“同一个节点在多个树中出现”的一致性问题。
2. 现有 `reparent_task`、`insert_subtask_and_start`、统计重放和树高亮逻辑都默认每个任务只有一条父链和一个显示位置。

### 4.4 标签语法与存储层的关系

用户提出的 `#!task: xxx`、`#!box: xxx`、`#!others` 是有价值的，但更适合作为“输入语法”，不适合作为底层唯一存储方式。

推荐方案：

1. 存储层使用 `boxes` 实体和 `tasks.box_id` 外键。
2. 输入层可以支持：
   - `#!task:<box-name>`
   - `#!box:<box-name>`
   - `#!others`
3. 若后续需要兼容命令输入，初版应使用“无空格”的规范写法；当前命令解析器无法直接正确支持 `#!task: xxx` 这种带空格语法。
4. 普通标签仍然保留，用于次级分类、筛选和统计；box 不与普通标签混存为同一语义层。

### 4.5 Box 状态

建议 box 具有独立状态，而不是从任务状态推导：

1. `active`
   - 默认可见，可继续使用。
2. `completed`
   - 默认自动隐藏，但可在筛选中显示。
3. `archived`
   - 长期隐藏，仅管理视图可见。

建议规则：

1. box 标记为 `completed` 是显式用户操作，不是根据任务自动推断。
2. 若 box 内仍存在 `running` 或 `paused` 任务，标记完成前应阻止或要求先结束这些任务。
3. 系统内置 `others` box 不建议支持“完成”语义。

### 4.6 任务管理型 Box 的特殊视图 a

这里必须单独说明：

当前系统并不具备甘特图所需的核心计划字段。若要做真正可用的甘特图，至少还需要：

1. 任务生命周期状态，例如 `todo / doing / done / blocked`。
2. 计划开始时间。
3. 计划结束时间或截止时间。
4. 预计时长。
5. 可选的依赖关系。

因此，建议把“任务管理型 box 的特殊视图 a”拆成两步：

1. 第一版先实现“项目视图占位版”，展示平铺任务列表、状态统计、进行中任务、总耗时、最近活动。
2. 只有在任务规划字段明确后，再决定是否升级为甘特图或其它项目视图。

## 5. 功能需求

### FR-BOX-001 Box 管理

系统应支持：

1. 创建 box。
2. 指定 box 类型为 `project/category/others`。
3. 重命名 box。
4. 调整 box 排序。
5. 标记 box 完成。
6. 恢复已完成 box。
7. 归档 box。

验收标准：

1. 用户可以在新的 box 页面查看全部 box。
2. 已完成 box 默认不显示在主列表中。
3. 用户可以切换“显示已完成 box”。

### FR-BOX-002 Box 切换与任务树切换

系统应支持：

1. 用户选择某个 box 后，任务树只展示该 box 的任务。
2. 当前 box 切换后，树搜索、展开状态、选中状态都基于该 box 独立维护。
3. 当前 active task 所属 box 应可被快速定位。

验收标准：

1. 切换 box 后，任务树不会混入其他 box 的根节点。
2. 同一任务不会在多个 box 的树中重复出现。

### FR-BOX-003 任务与 Box 的绑定

系统应支持：

1. 创建任务时指定 box。
2. 子任务自动继承父任务 box。
3. 将任务子树整体移动到另一个 box。
4. 在同 box 内执行 `reparent`。
5. 自动补全不会出现其他 box 的任务，输入其它box的任务是无效的。

验收标准：

1. 普通 `reparent_task` 不会制造跨 box 父子关系。
2. 移动任务子树到新 box 后，整棵子树归属一致。

### FR-BOX-004 新的 Box 页面

新页面信息架构：

1. 左侧：box 列表。
2. 右侧：当前 box 内容区域。

右侧内容规则：

1. 对 `category` box：
   - 展示该 box 的任务列表或根任务列表。
   - 默认不展开子任务。
   - 可显示任务标题、计时状态、Ex/In 时长、子任务数、最近活动时间。
2. 对 `project` box：
   - 展示项目视图 a。
   - 第一阶段先提供占位版项目总览，不直接实现甘特图。
3. 对 `others` box：
   - 展示更轻量的任务列表和快速新建入口。

### FR-BOX-005 杂项 Box 与临时任务

系统应支持“低结构负担”的快速计时流程：

1. 用户进入 `others` box 后，可以一键创建并开始一个根任务。
2. 临时任务不要求先挂到某个项目树下。
3. 临时任务停止后仍保留记录，便于统计和回顾。
4. 后续可把临时任务移动到普通 box 或项目 box。

说明：

这一能力的重点不是“任务不存在”，而是“临时任务不需要先被纳入复杂树结构”。本质上仍是任务，只是属于杂项工作空间。

### FR-BOX-006 任务管理型 Box 的项目视图

第一阶段最小要求：

1. 展示该 box 的根任务列表。
2. 展示每个根任务的计时状态、累计耗时、子任务数、最近活动。
3. 展示 box 总耗时、进行中任务数、最近活跃任务。

第二阶段扩展要求：

1. 为项目任务引入独立的项目状态字段。
2. 引入计划字段后，评估甘特图或看板视图。

### FR-BOX-007 Box 完成后的隐藏规则

系统应支持：

1. 用户手动将 box 标记为完成。
2. 完成后的 box 默认从主 box 列表隐藏。
3. 已完成 box 内的历史任务和统计数据仍可被查看。

### FR-BOX-008 统计维度

系统应支持：

1. 统计页查看“全部 box”。
2. 统计页按指定 box 过滤。
3. `project` box 的项目视图使用 box 过滤后的统计结果。

## 6. 数据模型与存储层影响

### 6.1 推荐模型

建议新增：

1. `boxes`
   - `id TEXT PRIMARY KEY`
   - `name TEXT NOT NULL`
   - `box_type TEXT NOT NULL CHECK(box_type IN ('project', 'category', 'others'))`
   - `status TEXT NOT NULL CHECK(status IN ('active', 'completed', 'archived'))`
   - `sort_order INTEGER NOT NULL DEFAULT 0`
   - `created_at INTEGER NOT NULL`
   - `completed_at INTEGER`
   - `archived_at INTEGER`
2. `tasks.box_id`
   - `TEXT REFERENCES boxes(id)`

建议索引：

1. `idx_tasks_box_id`
2. `idx_boxes_status_sort_order`
3. 如项目视图需要，可增加 `idx_tasks_box_parent`

### 6.2 迁移策略

当前数据库只有全局任务树，因此迁移时需要为历史任务提供默认落点。

推荐迁移策略：

1. 新增内置 `others` box。
2. 另外新增一个系统默认 box，例如 `未分箱` 或 `迁移任务`。
3. 将现有全部任务回填到该默认 box，而不是直接丢进 `others`。
4. 后续用户可手动将历史任务拆分到不同 box。

原因：

1. 现有历史任务并不都属于“杂项”。
2. 若全部迁入 `others`，会污染用户未来用于临时任务的工作空间。

### 6.3 事件日志影响

当前系统已把 `reparent`、`tag_add`、`tag_remove` 写入 `time_events`。为保持结构操作可追溯，建议新增事件类型：

1. `box_assign`
2. `box_move`

若第一阶段不记录也可运行，但会造成 box 结构变更不可追溯，与现有事件溯源思路不完全一致。

### 6.4 对项目视图 a 的延迟建模

若后续要真正实现甘特图，推荐新增而不是复用现有计时状态：

1. `workflow_state`
2. `planned_start_at`
3. `planned_end_at` 或 `due_at`
4. `estimate_seconds`

建议这些字段放在单独阶段引入，避免 box 第一阶段把范围做穿。

## 7. API 与后端服务影响

### 7.1 需要新增的查询/命令

建议新增：

1. `list_boxes()`
2. `create_box(name, box_type)`
3. `rename_box(box_id, name)`
4. `update_box_status(box_id, status)`
5. `move_task_subtree_to_box(task_id, target_box_id)`
6. `get_box_workspace(box_id)`
7. `get_box_summary(box_id, range?)`

### 7.2 需要扩展的现有接口

建议调整：

1. `TaskRecord`
   - 新增 `box_id`
2. `get_overview(range)`
   - 增加可选 `box_id`
   - 若不传 `box_id`，保持当前“全局视图”兼容行为
3. `get_focus_summary(range)`
   - 增加可选 `box_id`
4. `create_task(title, parent_id, box_id?)`
   - 根任务创建时支持显式指定 box
   - 子任务创建时可忽略 `box_id` 并从父任务继承
5. `reparent_task(task_id, new_parent_id)`
   - 增加同 box 校验
6. `insert_subtask_and_start(parent_task_id, title)`
   - 子任务自动继承父任务 box

### 7.3 兼容原则

为避免一次性破坏前端，建议遵循以下兼容策略：

1. 旧页面调用 `get_overview()` 不传 `box_id` 时，仍得到全局任务集。
2. 新页面和新树视图开始显式使用 `box_id`。
3. 等 box UI 稳定后，再逐步把老的“全局树视图”降级为管理模式或调试模式。

## 8. 前端与交互影响

### 8.1 `任务树` 页面

当前树页面默认展示所有根任务。引入 box 后，建议：

1. 页面顶部新增当前 box 切换器。
2. 树只展示当前 box 的任务。
3. 快速创建任务默认创建到当前 box。
4. 批量删除、搜索、展开状态都按当前 box 独立处理。

### 8.2 `任务详情` 主工作台

当前 `/` 页面右侧 mini 树基于“当前任务系”的根节点。引入 box 后，建议：

1. 增加当前任务所属 box 的可视标识。
2. mini 树仍可围绕当前任务所在根节点展示，但必须保证节点只来自当前任务所属 box。
3. 当活动任务位于隐藏 box 或已完成 box 时，界面应给出可返回该 box 的入口。

### 8.3 `统计摘要` 页面

建议：

1. 新增 box 过滤器。
2. 默认支持“全部 box / 当前 box / 指定 box”三种范围。
3. 项目型 box 的专属视图应复用相同统计基础，而不是再造一套口径。

### 8.4 `命令模式` 与输入语法

当前命令系统需要重点调整：

1. 目前仅支持 `#tag` 自动补全。
2. `#!task: xxx` 与 `#!box: xxx` 需要新的解析规则。
3. 初版若不改命令语法，建议 box 绑定由 UI 完成，命令模式后续补齐。
4. 若要在第一阶段支持 box 语法，建议使用不含空格的写法，例如 `#!task:project-a`。

### 8.5 `文档页`

需要补充：

1. box 类型说明。
2. box 切换说明。
3. box 绑定语法说明。
4. 临时任务与 `others` box 的使用说明。

## 9. 与现有功能的关系和影响

### 9.1 会被保留的核心约束

以下现有约束应保持不变：

1. 单活动上下文。
2. 子任务插入时父任务自动暂停，子任务结束后父任务自动恢复。
3. 任务树仍使用邻接表 `parent_id`。
4. 计时统计仍以事件流重放为真相。

### 9.2 会被修改的核心边界

以下能力会发生设计层变化：

1. “全局唯一任务树”变为“按 box 切换的任务树”。
2. “当前任务系”不再是唯一的空间分组方式。
3. 标签不再承担工作空间语义，box 才是主导航维度。
4. 项目管理能力不能再只依赖 `status=running/paused/...`。

### 9.3 风险点

1. 若只用标签模拟 box，会出现命名冲突、状态无法管理、查询复杂、类型不稳定的问题。
2. 若允许一个任务归属多个 box，会显著破坏当前树、重挂载、统计和命令体系。
3. 若过早承诺甘特图，会立即牵出计划字段、项目状态、依赖关系等额外范围。
4. 若直接把全部历史任务迁入 `others`，会让杂项 box 失去“低负担临时空间”的价值。

## 10. 分阶段实施计划

### 阶段 0：需求冻结与决策 Gate

目标：先锁死语义，避免实现后返工。

需要明确：

1. `others` 是否为唯一系统内置杂项 box。
2. box 是否允许改类型。
3. 项目视图 a 的第一版长什么样。
4. 是否第一阶段就支持 `#!task:<name>` 命令语法。

完成标准：

1. 本文档评审通过。
2. box 类型、状态和绑定规则冻结。
3. 项目视图 a 的“第一版不是甘特图”被明确接受。

### 阶段 1：Box 基础设施

目标：把 box 作为一等实体落库，并让后端具备 box 感知能力。

工作项：

1. 新增 `boxes` 表和 `tasks.box_id`。
2. 添加迁移和历史任务回填。
3. 扩展 `TaskRecord`。
4. 增加 `list_boxes / create_box / rename_box / update_box_status / move_task_subtree_to_box`。
5. 给 `get_overview / get_focus_summary` 增加 `box_id` 过滤。
6. 给 `create_task / reparent_task / insert_subtask_and_start` 加入 box 规则。

验收标准：

1. 任务可稳定落入 box。
2. 不会产生跨 box 父子关系。
3. 旧页面在不传 `box_id` 时仍能工作。

### 阶段 2：Box 感知的任务树与 Box 页面

目标：把“box 切换”真正带到 UI。

工作项：

1. 任务树页面增加 box 切换器。
2. 新建 box 页面，左侧 box 列表，右侧 box 内容区。
3. 类别 box 右侧先展示平铺任务列表，不展开子任务。
4. 完成 box 的默认隐藏和筛选。
5. 主工作台增加当前 box 展示和跳转入口。

验收标准：

1. 用户可以在 UI 中创建、切换、完成和恢复 box。
2. 切换 box 后，树视图和快速创建都正确落到当前 box。

### 阶段 3：杂项 Box 与临时任务体验

目标：让临时事务计时成为低摩擦流程。

工作项：

1. `others` box 提供“快速创建并开始”入口。
2. 增加把临时任务移动到其他 box 的入口。
3. 评估是否增加“临时任务清理/归档建议”。
4. 视情况补命令模式中的 box 绑定语法。

验收标准：

1. 用户可在 2 秒内从 `others` box 启动一个临时任务。
2. 临时任务不需要先进入复杂任务树才能开始计时。

### 阶段 4：项目型 Box 的专项视图 a

目标：把 `project` box 做成真正有价值的项目工作区。

工作项：

1. 定义项目状态字段。
2. 定义计划字段。
3. 先实现项目总览版视图：
   - 根任务进度列表
   - 进行中任务
   - 总耗时
   - 最近活动
4. 在字段成熟后，再评估甘特图或看板。

验收标准：

1. `project` box 的右侧视图与 `category` box 明显不同。
2. 项目视图不再仅仅是“另一种任务树显示方式”。

## 11. 建议的开发顺序与优先级

推荐优先级：

1. 先做 box 作为一等实体。
2. 再做按 box 切换的树和 box 页面。
3. 再做 `others` 的临时任务体验。
4. 最后做 `project` box 的特殊视图。

不推荐的顺序：

1. 先做甘特图，再回头补 box 模型。
2. 先把 box 直接做成标签，再补管理实体。
3. 先允许多 box 归属，再补约束。

## 12. 当前仍需确认的开放问题

1. `others` box 是单例系统 box，还是允许多个杂项 box。
2. box 名称是否要求同类型内唯一，还是全局唯一。
3. 用户是否需要“默认启动 box”。
4. 已完成 box 中若仍有未停止任务，产品是禁止完成还是自动停止后完成。
5. 命令模式中 box 绑定语法是否放入第一阶段。
6. 项目视图 a 的最小形态是“项目总览卡片”还是“项目任务表格”。

## 13. 最终建议

建议把这次需求正式拆成两个 Epic：

1. Epic A：Box 基础能力
   - box 实体
   - 任务归属
   - box 切换任务树
   - box 页面
   - 完成隐藏
   - 临时任务承载
2. Epic B：项目型 Box 专项视图
   - 项目状态模型
   - 计划字段
   - 项目总览视图
   - 甘特图可行性评估

只有这样，TimeFiles 才能在不破坏现有计时核心的前提下，把“任务记录工具”向“可用的个人项目管理工具”稳步推进。
